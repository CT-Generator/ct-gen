## Context

Two cross-cutting locale-correctness gaps remain after `nl-no-english-fallback`:

- `web/app/g/[id]/page.tsx` `generateMetadata` returns `{ title, description, openGraph: { title, description, images } }` with no `locale` field. Next.js merges metadata: when a child page declares `openGraph`, it overrides the parent layout's `openGraph` entirely (Next docs: "Metadata merging is shallow per top-level field"). So the layout-level `og:locale` set by `nl-no-english-fallback` is effectively erased for any page that overrides openGraph. Confirmed live across all three locales — `og:locale` is missing from `/g/<id>`.
- `web/app/imprint/page.tsx` and `web/app/privacy/page.tsx` render English JSX bodies. There's no locale switch, no dictionary lookup, no notice. The German imprint was DEFERRED in `multilingual-german` task 3.4 because legally-original German content needs to be authored against current German jurisdictional law (not auto-translated). Dutch was never scheduled.

Both surfaces are user-facing, so the just-shipped "no English fallback for non-en locales" rule applies — but neither was actually fixed.

## Goals / Non-Goals

**Goals:**
- Per-`/g/[id]` share previews show the row's locale (`og:locale`) and a description in that locale.
- `/de/imprint`, `/nl/imprint`, `/de/privacy`, `/nl/privacy` render a visible "translation pending" notice in the active locale, then continue to render the English body until a jurisdictional original exists.
- Internationalization spec acknowledges the legal-page exemption explicitly so the no-fallback rule and the deferred-legal-authoring reality stop contradicting each other.

**Non-Goals:**
- Author the German + Dutch jurisdictional originals. That requires legal expertise; out of scope for a copy/UI change. The notice flags the gap so it stays visible.
- Localize the OG IMAGE (`/api/og/[id]`). The image is generated by Satori and currently shows the recipe glyphs + brand + culprit/event/motive triple — those values are already locale-correct (the row's persisted values). Image-side localization (e.g., German caption text) is a future concern; this change covers only the meta-tag side.
- Touch the page-content `description` and `title` strings beyond what's needed for the locale fix. The OG description for `/g/[id]` becomes locale-keyed; everything else in the metadata stays as-is.

## Decisions

### `/g/[id]` `og:locale` mapped via Record<Locale, string>

The metadata function already loads the row (`gen.locale`) before returning. Add a `Record<Locale, "en_US" | "de_DE" | "nl_NL">` map and set `openGraph.locale`. This mirrors the layout-level fix from `nl-no-english-fallback`. Compile-time: extending `Locale` later forces every consumer of the map to provide a value.

### `/g/[id]` `og:description` keyed by row locale

Coin a dictionary key `meta.og_description_generation` (en/de/nl). Look up via `getDict(rowLocale).meta.og_description_generation`. Same data travels into `twitter.description` (which is currently identical to `openGraph.description`).

**Alternative considered:** keep the description English everywhere (the brand is anglophone). Rejected — the persona review confirmed that a German visitor sharing a German theory wants the preview in German; the cost is one dictionary key per locale.

### Imprint / privacy translation-pending notice

Approach:
- `imprint/page.tsx` and `privacy/page.tsx` become locale-aware: read `await readLocale()`, render a small notice block above the existing English body when `locale !== "en"`, then continue rendering the English JSX as before.
- The notice has two parts: a heading (e.g., "Translation pending" / "Übersetzung in Arbeit" / "Vertaling in voorbereiding") and a one-sentence explanation in the active locale ("This page's legally-required content has not yet been authored in <German|Dutch>. The English source follows.").
- New dictionary section `legal` with two keys: `translation_pending_h` and `translation_pending_body`. Both keys exist in en for type-completeness, but the EN page never renders the notice (gated on `locale !== "en"`).

**Alternative considered:** render only the notice and hide the English body. Rejected — the page still has a legal function (German/Dutch operators need *some* content to point users to). Until originals exist, English-with-notice is more useful than blank-with-notice.

**Alternative considered:** redirect `/de/imprint` and `/nl/imprint` to `/imprint`. Rejected — the `cgen_lang` cookie + URL-prefix logic should not be subverted by per-route redirects; the visitor stays on their locale path and sees the gap acknowledged.

### Internationalization spec exemption scenario

Add a new requirement in `internationalization`:
> "Legally-significant page bodies (imprint, privacy) are subject to separate jurisdictional-original authoring requirements. Until a jurisdiction-original exists for a given locale, the page MAY render the English source body PROVIDED a localized 'translation pending' notice renders above the body in the active locale. Rendering the English body without the notice is a violation of the no-English-fallback rule."

This explicitly narrows the no-fallback rule's reach and gives the imprint/privacy team a clear bar to clear.

### Dutch translation-pending notice in pass-1 Dutch

Mark the new `legal.translation_pending_*` Dutch values with the standard `// FIXME: pass 2` per project convention. Pass-1 wooden literal is allowed.

## Risks / Trade-offs

- **[Layout `og:locale` gets shadowed by per-page openGraph]** → This is the bug we're fixing on `/g/[id]`, but it could lurk on any future page that overrides openGraph. Add a project convention note in design.md (recorded here) that every page-level `generateMetadata` returning an `openGraph` block MUST include `locale`. Not a spec rule per se — a convention surfaced in code review.
- **[Translation-pending notice clutters legal pages]** → Two extra lines at the top of an already-light page. Minor visual cost; the alternative (silent English fallthrough) is the worse trade.
- **[Dutch FIXME copy]** → Pass-1 wooden literal; will need pass-2 native review like the rest of `nl.ts`. Acceptable per existing project posture.
- **[OG description regression risk for English]** → Current English description is "Made with the four-move recipe. Pick an event, a culprit, a motive — watch the theory build itself." We carry this verbatim into `meta.og_description_generation` for English; no change visible to English shares.
